Early-Stage Testing Philosophy

When testing early-stage, in-progress work, your focus is fundamentally different from late-stage testing. Understanding this distinction will help you get more useful feedback.

Note: These are general project categories. Your work may incorporate aspects of multiple types, and it is fine to mix testing methods to fit your specific project.

DO Test Early On:

  • Games: Core concepts, fundamental interactions, challenge comprehension
  • Installations: Spatial understanding, interaction discovery, cause-and-effect, social dynamics
  • Static/Motion Media: Visual hierarchy, message comprehension, pacing, readability
  • UX/Apps: Task comprehension, navigation clarity, mental model alignment, action affordances

DON'T Worry About (Yet):

  • Visual polish or final aesthetics (unless testing visual communication specifically)
  • Complete content or all features
  • Performance optimization
  • Edge cases or rare scenarios
  • Perfect balance or tuning
  • Final materials or fabrication quality
Methods for Developing Questions

Several structured methods can help you identify what questions to ask:

Critical Assumption Inventory

List all assumptions about how users will interact, rank by risk (critical, important, minor), then convert each to a testable question.

Isolated Variable Method

Test ONE thing at a time by creating minimal prototypes that isolate specific elements (e.g., test form separately from interaction model).

Progressive Disclosure Test

Test concept, then rough interaction, then refined version in sequence with the same users to see where understanding develops.

Core Loop Extraction

Identify the fundamental repeating interaction and test only that loop (Approach, Trigger, Feedback, Understand).

Wizard of Oz Testing

Test interactions not yet implemented by manually creating responses while users think it is automatic.

Task-Based Flow Analysis (UX/Apps)

Identify top user tasks, map ideal flows, find potential failure points, write questions for each failure point.

Spatial/Temporal Analysis (Installations/Motion)

Map approach sequences or viewing timelines and test at each critical distance or timestamp.

Error Path Testing (UX/Apps)

Intentionally create error scenarios and watch what users do when stuck to learn about error prevention and recovery.

Testing Different Prototype Fidelities

Paper Prototypes

Best for: Layout, spatial relationships, sequence, information hierarchy

  • Installations: Cardboard boxes, foam core, tape for floor markings
  • Visual Media: Hand sketches at correct size, sharpie wireframes
  • Motion: Storyboard frames, index cards flipped at speed
  • UX/Apps: Draw screens on paper, Post-its for interactive elements

Digital Wireframes/Mockups

Best for: Layout refinement, timing, style exploration, navigation testing

  • Installations: Video mockups, photo composites, simple Unity visualization
  • Visual Media: Figma wireframes, style tiles, component tests
  • Motion: After Effects animatic, style frames with timing
  • UX/Apps: Linked Figma screens, basic tap interactions

Interactive Prototypes (Low-Fi)

Best for: Sensor responsiveness, gesture testing, complete user flows

  • Installations: Arduino + simple sensor, one working interaction
  • Motion: One sequence polished, key section animated fully
  • UX/Apps: Clickable Figma prototype, basic web prototype

Physical Mockups & Scale Models (Installations)

Critical for spatial work - human body proportions only make sense at 1:1

  • PVC pipe frames, cardboard and hot glue
  • Floor tape for positioning
  • Test in actual intended space if possible
  • 1:10 or 1:20 models for overall spatial planning
Common Question Mistakes
Mistake Why It's Bad Better Alternative
"Do you like this?" Subjective preference, not useful "What does this communicate to you?"
"Is the navigation clear?" Yes/no, assumes there is clear nav "How would you get to [section]?"
"Don't you think this is intuitive?" Leading question "What would you try first?"
"Was that too fast?" Leading toward one answer "How was the pacing?"
"Is this easy to understand?" Leading with "easy" "What is this telling you?"
"Click here to go to settings" Giving instructions "How would you change your password?"
Before Your Session

Installation Testing

  • Test mockup is stable and will not fall
  • Space is clear for movement
  • Lighting is appropriate (or test lighting variations)
  • Wizard of Oz responses are ready (if needed)
  • Camera positioned to capture approach and interaction
  • Space allows testing at multiple distances

Visual Media Testing

  • Printed at actual size/scale
  • Can be viewed at intended distance
  • Multiple versions ready (if testing comparisons)
  • Lighting matches final context (screen vs. print vs. projection)

Motion Media Testing

  • Video plays smoothly (no tech difficulties)
  • Can show at different speeds if needed
  • Can pause/replay easily
  • Viewing screen/projection size is appropriate
  • Sound is clear (if testing with audio)

UX/App Testing

  • Prototype works (or Wizard of Oz is ready)
  • Tested on actual device (not just desktop)
  • Specific tasks written out (not just "explore")
  • Know which features work and which do not
  • Screen recording software ready (if remote)
  • Can reset prototype between tests
Final Reminders

The Core Principle

Test spatial relationships, visual hierarchy, sequence clarity, and navigation structure before worrying about materials, polish, or final execution.

The Permission Slip

Cardboard installations, sharpie sketches, flipbook animatics, and paper app prototypes are not just acceptable - they are often better for testing than polished prototypes.

The Reality Check

If 3+ people do not understand your spatial relationship / visual hierarchy / sequence / navigation structure, it is not the execution - it is the concept.

The UX Reminder

Users will complete tasks the "wrong" way. That is not wrong - that is your design failing to match their mental model. Change your design, not your users.
