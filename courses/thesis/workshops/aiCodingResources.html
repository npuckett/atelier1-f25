AI Coding Resources

This page covers the foundational concepts for working with AI coding agents effectively. The key insight: the same documentation that helps a human collaborator understand your project will help an AI agent understand it. There is no special "AI documentation"—just good documentation.

Note: You do not need to use AI coding agents to benefit from clear project documentation. These practices are valuable for any development workflow.

Why Documentation Matters

Good documentation clarifies your thinking before you write code, creates shared understanding with collaborators, and serves as reference when you return to work months later.

For Human Developers:

  • Forces you to think through scope before building
  • Prevents misunderstandings in team projects
  • Makes it easier to pick up where you left off

For AI Coding Agents:

  • Provides essential context for relevant code generation
  • Reduces back-and-forth clarification cycles
  • Helps agents understand constraints and preferences
  • Produces more consistent output on first attempt
Industry Standard Document Types

Professional software development uses several types of documentation. You do not need all of these for every project—choose what fits your scope.

README File

The front door to your project. Explains what it does, how to set it up, and how to use it.

When to use: Every project, regardless of size.

Product Requirements Document (PRD)

Describes what you are building and why, without specifying how. For creative coding projects -- games, AR experiences, generative visuals, projection mapping, interactive installations -- this means describing the experience, interactions, and behaviors rather than implementation details. Written as a Markdown (.md) file.

When to use: Any project with enough complexity that you need to think through what you are building before you build it. This is the focus of the workshop.

Software Requirements Specification (SRS)

Detailed, formal document describing exactly what the software should do. Based on IEEE 830 standard.

When to use: Formal projects, client work with contracts, or when requirements must be unambiguous.

Technical Design Document (TDD)

Describes how you will build the system—architecture, technologies, data structures, implementation approach.

When to use: When implementation approach is not obvious, or when documenting technical decisions for a team.

Project Brief / Project Specification

Concise document combining elements of PRDs and technical specs. Practical for smaller projects or individual work.

When to use: Thesis projects, personal projects, hackathons—this is the recommended starting point for this workshop.

Choosing the Right Level
Project Type Recommended Documentation
Quick prototype / learning project README only
Thesis project component README + Project Brief
Team project / hackathon README + lightweight PRD
Client work / freelance README + PRD + Technical Design
Best Practices for Documentation

Use Clear, Hierarchical Structure

Organize information in logical sections with headers. Both humans and AI agents find relevant context faster when structure is clear.

Be Explicit About Constraints

Do not assume anything is "obvious." State constraints clearly.

❌ Vague: "The installation should be responsive"

✓ Explicit: "Sensor must detect motion within 2 meters, response latency under 100ms"

Use Consistent, Structured Formats

When listing features or requirements, use the same format throughout. Tables, JSON schemas, or consistent bullet formats make parsing reliable.

Separate What from How

Keep requirements (what) separate from implementation details (how) unless you have strong technical constraints.

❌ Mixed: "Use p5.js to create a particle system that responds to sound"

✓ Separated: "Requirement: Visual system responds to audio input. Technical preference: p5.js (team familiarity)"

Include Examples and Edge Cases

Concrete examples prevent misunderstandings. What happens when there is no input? When input is extreme? When multiple users interact?

Version and Date Your Documents

Requirements change. Include when the document was created/updated to prevent confusion.

Project Brief Template

Use this template as a starting point for your PRD. Not every section applies to every project—adapt as needed. 

# [Project Name]

**Version:** 1.0
**Date:** [Date]
**Author:** [Your name]

## 1. Project Overview

### What Is This?
One paragraph describing what you are building.

### Experience Description
What does someone see/hear/feel when they encounter this?
Describe the experience from the audience/player/user perspective.

### Who Is It For?
Who will use or experience this? What context are they in?

## 2. Interaction Model

### How Does It Work?
Describe the primary interaction. What does the user do,
and what happens in response?

### Key States
What are the distinct modes or states of the experience?
| State | What Happens | Transitions To |
|-------|-------------|----------------|
| [e.g., Idle] | [What the system does] | [Next state] |

## 3. Features and Requirements

### Core Features (Must Have)
| Feature | Description | Priority |
|---------|-------------|----------|
| [Feature 1] | [What it does] | High |

### Nice-to-Have Features
| Feature | Description | Priority |
|---------|-------------|----------|
| [Feature 1] | [What it does] | Medium |

### Out of Scope
What are you explicitly NOT building?

## 4. Visual and Audio Requirements

### Visual Style
Describe the look: colors, mood, references, aesthetic.

### Audio
What sounds exist? Music? Ambient? Reactive?

### Performance Targets
- [e.g., Must run at 60fps]
- [e.g., Must support 1920x1080 projection]

## 5. Technical Approach

### Technology Stack
- **Platform:** [e.g., Web, Unity, Arduino, Unreal]
- **Languages:** [e.g., JavaScript, C#, Python]
- **Key Libraries:** [e.g., p5.js, Three.js, Shader Graph]

### Technical Constraints
- [e.g., Must run in browser without plugins]
- [e.g., Must work with existing sensor hardware]
- [e.g., Must build for Windows standalone]

## 6. Success Criteria
How will you know it works?
- [ ] [Specific, testable criterion]
- [ ] [e.g., Player can complete one full interaction loop]
- [ ] [e.g., Runs at target framerate on target hardware]

## 7. Open Questions
What have you not figured out yet?
Agent-Assisted Documentation

If you have a rough idea but are not sure how to structure it, you can use an AI agent to help organize your thoughts. This is a collaborative process—you provide raw material, the agent helps structure it.

Step 1: Brain Dump

Write down everything you know in plain language. Include what you want to build, who it is for, features you have thought of, technical preferences, and uncertainties.

Step 2: Ask the Agent to Structure It

Example prompt:

"I have a rough idea for a project. Can you help me turn this into a structured project brief? Here is what I am thinking: [paste brain dump]. Please organize this with: a problem statement, core features vs nice-to-have, technical approach, and questions I should answer before starting."

Step 3: Review and Refine

The agent will produce a draft. Verify it captures what you meant, fill gaps, add detail, and remove incorrect assumptions. You know your project better than it does.

Step 4: Iterate

Use follow-up prompts to refine specific sections. Ask the agent to identify gaps or present options with tradeoffs.

Useful Prompts for Documentation

When your ideas are unstructured:

"I have these rough notes about my project. Can you: 1) Identify what is clear vs what needs more detail, 2) Suggest a structure that would make this easier to understand, 3) List questions I should answer to complete the documentation."

When you want the agent to identify gaps:

"Review this project brief and tell me: What assumptions am I making that I should state explicitly? What edge cases have I not considered? What technical decisions am I deferring that might cause problems later?"

When you want options presented:

"For [feature], I am not sure about the best approach. Can you present 2-3 options with: how each would work, pros and cons, and which you would recommend for someone new to this. Do not decide for me—just lay out the options clearly."

When you need a specific format:

"Convert this project description into a structured format with: a features table (columns: Feature, Description, Priority), data entities in JSON format, and user stories in 'As a [user], I want [goal], so that [reason]' format."

Agent Instruction Files

Agent instruction files are configuration files that tell AI coding agents how to behave in your specific project. Think of them as "onboarding documents" for your AI assistant.

Note: This section is optional. Instruction files are useful if you are working with coding agents regularly, but not required for the workshop.

Example: 3D Movement System

This example shows how a thesis student might document a core game mechanic. Notice that it focuses on how the movement should feel and behave, not on implementation code. 

# 3D Character Movement System

**Version:** 1.0
**Date:** March 2026
**Author:** [Student Name]

## What Is This?
A third-person character movement system for a 3D
exploration game built in Unity. The movement needs to
feel responsive and satisfying -- the player spends the
entire game moving through space, so this has to be right.

## Experience Description
The player controls a character moving through large,
vertical environments. Movement should feel grounded and
weighty but not sluggish. Jumping should feel generous
(forgiving input timing). The camera follows behind the
character with some lag.

## Interaction Model

### Input
- Left stick / WASD: movement relative to camera
- A button / Space: jump
- Left trigger / Shift: sprint

### Key States
| State | What Happens | Transitions To |
|-------|-------------|----------------|
| Idle | Character stands, small breathing animation | Walk, Jump |
| Walk | Acceleration curve to max speed over 0.2s | Idle, Sprint, Jump, Fall |
| Sprint | 1.8x walk speed, wider camera FOV | Walk, Jump, Fall |
| Jump | Upward force, reduced air control (70%) | Fall |
| Fall | Gravity increases over time (fast fall) | Land |
| Land | Brief recovery (0.1s), squash animation | Idle, Walk |

## Core Features
| Feature | Description | Priority |
|---------|-------------|----------|
| Ground movement | Acceleration/deceleration curves | High |
| Jump | Variable height based on button hold | High |
| Coyote time | 0.15s grace period after leaving edge | High |
| Sprint | Hold to run faster, release to slow | High |
| Slope handling | Adjust speed on inclines, slide on steep | Medium |
| Input buffering | Queue jump input 0.1s before landing | Medium |

## Out of Scope
- Wall running or climbing (separate system later)
- Swimming or other special movement modes
- Multiplayer synchronization
- Mobile/touch controls

## Visual and Audio Requirements

### Visual
- Squash/stretch on landing proportional to fall time
- Dust particles on hard landings and direction changes
- Sprint adds subtle camera shake and FOV increase

### Audio
- Footstep sounds vary by surface material
- Wind sound layer during sprint
- Landing impact sound scales with fall distance

### Performance Targets
- Must run at 60fps on mid-range hardware
- Movement physics at fixed timestep (FixedUpdate)

## Technical Approach
- **Platform:** Unity (Windows build)
- **Language:** C#
- **Input:** New Input System package
- **Physics:** CharacterController (not Rigidbody)

## Success Criteria
- [ ] Character accelerates/decelerates smoothly
- [ ] Jump height varies with button hold duration
- [ ] Coyote time works reliably at ledges
- [ ] Landing recovery feels snappy, not sticky
- [ ] Slopes do not cause jittering
- [ ] Runs at 60fps in test environment
Example: Simple UX -- Portfolio Website

This example shows how a student might document a straightforward web project. Notice that even a "simple" site benefits from clear descriptions of layout, navigation, and content structure. 

# Portfolio Website

**Version:** 1.0
**Date:** March 2026
**Author:** [Student Name]

## What Is This?
A single-page portfolio website to present thesis work
and selected projects. The site needs to load fast, work
on mobile, and feel minimal -- the work itself should be
the focus, not the website design.

## Experience Description
A visitor lands on the page and sees a brief introduction
and a grid of project thumbnails. Clicking a thumbnail
opens a detail view with images, a short description, and
a link to the full project. The site uses smooth scroll
navigation between sections. No login, no CMS, no blog --
just a clean showcase.

## Interaction Model

### Navigation
- Fixed top nav bar with section links (About, Work, Contact)
- Clicking a nav link smooth-scrolls to section
- Nav bar highlights the current section on scroll

### Project Grid
- Thumbnails arranged in a responsive grid
- Hover: thumbnail dims slightly showing project title
- Click: opens a detail overlay or navigates to detail section

### Key States
| State | What Happens | Transitions To |
|-------|-------------|----------------|
| Landing | Hero section with name and one-line bio | Scroll or nav click |
| Browsing work | Grid of project thumbnails visible | Click thumbnail |
| Project detail | Overlay with images, description, links | Close overlay or back |
| Contact | Simple contact form or email link | Form submit |

## Core Features
| Feature | Description | Priority |
|---------|-------------|----------|
| Responsive layout | Works on phone, tablet, desktop | High |
| Project grid | Thumbnail grid with hover states | High |
| Project detail view | Overlay or section with full project info | High |
| Smooth scroll nav | Section-based navigation with scroll | High |
| Contact section | Email link or simple contact form | Medium |
| Dark/light toggle | Switch between color modes | Low |

## Out of Scope
- Content management system or admin panel
- Blog or news section
- User accounts or authentication
- E-commerce or payment processing

## Visual and Audio Requirements

### Visual Style
- Minimal, mostly white space with one accent color
- Typography-driven: large headings, clean body text
- Project images are the primary visual element
- No decorative illustrations or background patterns

### Audio
- None

### Performance Targets
- Page load under 2 seconds on 3G connection
- All images lazy-loaded and optimized (WebP)
- Lighthouse performance score above 90

## Technical Approach
- **Platform:** Static website (hosted on GitHub Pages)
- **Languages:** HTML, CSS, JavaScript
- **Key Libraries:** None required (vanilla JS)
- **Build:** None -- plain files, no bundler needed

## Success Criteria
- [ ] Site loads and displays correctly on Chrome, Safari, Firefox
- [ ] Layout works at 320px, 768px, and 1440px widths
- [ ] All project thumbnails link to correct detail views
- [ ] Nav highlights update correctly on scroll
- [ ] Lighthouse accessibility score above 90
- [ ] All images have alt text
Example: Interactive Art Installation

This example shows how a student might document a creative coding or installation project. Notice the emphasis on describing the experience and sensory qualities rather than technical implementation. 

# Tide Pool -- Generative Projection Installation

**Version:** 1.0
**Date:** March 2026
**Author:** [Student Name]

## What Is This?
A floor-projected interactive installation where
projected visuals respond to people walking through the
space. Abstract organic forms -- inspired by tide pools
and bioluminescence -- grow, drift, and react to presence
and movement. Built with p5.js and a depth camera.

## Experience Description
Visitors enter a darkened room with a large floor
projection (roughly 3m x 3m). When no one is present,
slow-moving abstract shapes drift across the surface like
organisms in still water. As a person steps into the
projection area, forms gather near their feet and pulse
outward in rings. Multiple people create overlapping
ripple patterns. When everyone leaves, the system
gradually returns to its idle drift over about 30 seconds.

## Interaction Model

### Input
- Depth camera (Kinect v2) mounted overhead, pointed down
- Tracks position and movement speed of up to 4 people
- No touch, no buttons, no explicit interaction required

### Behavior
- Presence triggers attraction: shapes migrate toward
  detected positions
- Movement speed affects energy: walking fast creates
  larger, brighter ripples; standing still creates gentle
  pulsing clusters
- Proximity between people blends their ripple fields

### Key States
| State | What Happens | Transitions To |
|-------|-------------|----------------|
| Idle | Slow organic drift, muted colors, no interaction | Active (person detected) |
| Active | Shapes attract to people, ripples pulse outward | Multi (2+ people), Idle (all leave) |
| Multi | Overlapping fields blend, colors shift and mix | Active (one leaves), Idle (all leave) |
| Fade-out | Energy slowly dissipates over 30 seconds | Idle |

## Core Features
| Feature | Description | Priority |
|---------|-------------|----------|
| Presence detection | Track up to 4 people via depth camera | High |
| Attraction behavior | Shapes migrate toward detected positions | High |
| Ripple generation | Movement creates expanding ring patterns | High |
| Idle animation | Ambient drift when no one is present | High |
| Speed responsiveness | Faster movement = larger/brighter ripples | Medium |
| Multi-person blending | Overlapping fields create mixed colors | Medium |
| Fade-out transition | Gradual return to idle after people leave | Medium |

## Out of Scope
- Sound reactivity (potential future addition)
- Hand or gesture tracking (full body only)
- Remote or web-based viewing
- Recording or playback of sessions

## Visual and Audio Requirements

### Visual Style
- Dark background, shapes in cool blues and greens
- Bioluminescent quality: soft glow, no hard edges
- Forms are organic and rounded, never geometric
- Color palette shifts slowly over time (cycle of
  about 5 minutes) to keep the experience evolving

### Audio
- Ambient generative drone that shifts with activity
  level (louder/more complex when active, quieter at
  idle) -- handled by a separate Max/MSP patch
  receiving OSC data from the visuals

### Performance Targets
- Must run at 60fps with 4 people tracked
- Projection resolution: 1920x1080 minimum
- Depth camera latency under 50ms

## Technical Approach
- **Platform:** Desktop application (macOS)
- **Language:** JavaScript
- **Key Libraries:** p5.js, Kinectron (depth camera bridge)
- **Audio:** Max/MSP receiving OSC messages
- **Display:** Short-throw projector, ceiling-mounted,
  pointed at floor

## Technical Constraints
- Computer must be within 5m of Kinect (USB cable)
- Room must be dark enough for projection to read clearly
- Kinect requires clear overhead view (no obstructions
  between camera and floor)

## Success Criteria
- [ ] Shapes visibly respond to a person within 0.5s
- [ ] Ripple scale changes based on movement speed
- [ ] System returns to idle within 30s of last person leaving
- [ ] Runs at 60fps with 4 simultaneous people
- [ ] Works in ambient light conditions of gallery space
- [ ] Installation runs unattended for 8 hours without crashing
Common Documentation Mistakes
Mistake Why It Is a Problem Better Approach
"Make it look good" Subjective, untestable Specify color palette, typography, reference images
"It should be responsive" Vague—responsive to what? "Response latency under 100ms after motion detected"
Mixing requirements and solutions Limits exploration of alternatives Separate what you need from how you might build it
No edge cases mentioned Agent will guess or ignore Explicitly state what happens in unusual situations
Assuming technical knowledge Misaligned expectations State your experience level and preferences
Documentation Checklist

Before starting to build (with or without an agent), can you answer:

The Basics

  • What problem am I solving?
  • Who is this for?
  • What does success look like?

Features

  • What are the must-have features?
  • What am I explicitly NOT building?
  • What are the nice-to-haves if I have time?

Technical

  • What technologies will I use?
  • What are my constraints?
  • What data do I need to store or process?

Clarity

  • Would a classmate understand what I am building?
  • Would an AI agent have enough context?
  • Are my requirements specific enough to test?

Remember

The Core Principle

Good documentation clarifies your own thinking. It happens to also make AI agents more useful, but that is a bonus.

The Permission Slip

It is okay to not know everything upfront. Document what you know, note what you are uncertain about, and iterate.

The Reality Check

If you cannot explain what you are building to a classmate, you are not ready to build it—with or without AI.
